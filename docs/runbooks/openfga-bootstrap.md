# Runbook — OpenFGA bootstrap

**Status:** Active. Phase 3.2 step 4.
**Source of truth:** [`docs/proposals/openfga.md`](../proposals/openfga.md) §3 step 4 ([Phase 3.2 — Stand up the sidecar](../proposals/openfga.md#phase-3-2-stand-up-the-sidecar)).
**Tracked in:** [`docs/implementation/phase-3-checklist.md`](../implementation/phase-3-checklist.md) Phase 3.2 step 4.

This runbook covers the **one-time-per-model-edit** bootstrap that wires Heimdall.Web to a freshly deployed OpenFGA sidecar. It does **not** cover Phase 3.4 step 7 (tuple-write hooks) or step 8 (backfill / reconciliation) — those have their own runbooks in a future PR.

## Prerequisites

- `fga` CLI installed:
  ```
  go install github.com/openfga/cli/cmd/fga@latest
  ```
- `jq` installed (any modern version).
- Network access from your shell to the private OpenFGA service. Either:
  - run from a Render shell on the `heimdall-ticket-tracker-web` service (private DNS reaches `heimdall-ticket-tracker-openfga:8080` automatically), **or**
  - tunnel through a VPN / SSH bastion that can reach Render's private network.
- The `OPENFGA_PRESHARED_KEY` value from the Render dashboard (auto-generated by `generateValue: true` on first deploy of the OpenFGA service).

## Step 1 — Create the `heimdall_authz` logical DB (one-time)

Render's blueprint (`render.yaml`) cannot provision a second logical DB on an existing Postgres instance. Do this **once** per environment, before the first deploy of the OpenFGA service:

1. Open a `psql` shell to the `heimdall-ticket-tracker-postgres` instance using the credentials from `DATABASE_URL`.
2. Run:
   ```sql
   CREATE DATABASE heimdall_authz OWNER heimdall;
   ```
3. Construct the `OPENFGA_DATASTORE_URI` by copying `DATABASE_URL` and substituting the database name segment with `heimdall_authz` (same host, port, user, password).
4. In the Render dashboard, set `OPENFGA_DATASTORE_URI` on the `heimdall-ticket-tracker-openfga` service to that value (it is declared `sync: false` in `render.yaml`).

## Step 2 — Deploy the blueprint

Push / sync the updated `render.yaml`. Render will:

1. Create or update the `heimdall-ticket-tracker-openfga` private service.
2. Run the `dockerCommand` wrapper: `openfga migrate` (creates / migrates the schema in `heimdall_authz`), then `exec openfga run`.
3. Generate `OPENFGA_AUTHN_PRESHARED_KEYS` on first deploy.

Wait for `/healthz` on the OpenFGA service to report healthy before proceeding.

## Step 3 — Run the bootstrap script

From a shell with network access to the sidecar:

```sh
export OPENFGA_API_URL='http://heimdall-ticket-tracker-openfga:8080'
export OPENFGA_PRESHARED_KEY='<paste from Render dashboard>'

./scripts/openfga-bootstrap.sh
```

The script:

- looks up an existing store named `heimdall`; creates one if absent (idempotent — never duplicates);
- writes `authz/model.fga` as a new model version (model IDs are immutable — every write returns a fresh `authorization_model_id`);
- prints `STORE_ID=…` and `AUTHORIZATION_MODEL_ID=…` to stdout;
- writes the same pair as JSON to `authz/.bootstrap-output.json` (gitignored).

## Step 4 — Update web service env vars

In the Render dashboard, on `heimdall-ticket-tracker-web`:

1. Set `OPENFGA_STORE_ID` to the printed `STORE_ID`.
2. Set `OPENFGA_AUTHORIZATION_MODEL_ID` to the printed `AUTHORIZATION_MODEL_ID`.
3. Set `OPENFGA_HEALTH_PROBE_ENABLED=true` (Phase 3.3 — opts the deployment into the fail-fast startup probe). Leave it unset (or `false`) only when you intentionally want the web service to boot without proving the sidecar is reachable (e.g. a one-off sidecar maintenance window).
4. Trigger a redeploy of the web service so the SDK picks them up at startup.

The web service's startup health probe (Phase 3.3 step 5) calls `ReadAuthorizationModel` against the pinned `AUTHORIZATION_MODEL_ID`, so it fails fast on either an unreachable sidecar or a model-id typo. If `OPENFGA_API_URL` / `OPENFGA_STORE_ID` / `OPENFGA_AUTHORIZATION_MODEL_ID` are partially unset, DI auto-falls-back to `NoOp*` services (deny-closed for checks, no-op for tuple writes) — useful for local dev, **never** what you want in production. The probe is your safety net: leaving it disabled in production while one of those env vars is missing means OpenFGA is silently inert.

## Step 5 — One-time tuple backfill (Phase 3.4)

After the first redeploy with the OpenFGA env vars set, the existing rows in `organizations` / `teams` / `projects` / `*_members` / `tickets` need their equivalent tuples written into the freshly-bootstrapped store. The `OpenFgaBackfillJob` does this idempotently:

1. In the Render dashboard, on `heimdall-ticket-tracker-web`, set `HEIMDALL_OPENFGA_BACKFILL=1`.
2. Trigger a redeploy. On boot, after the existing bootstrappers run, `OpenFgaBackfillRunner` invokes the backfill job, paginates the entire DB at ≤ 100 tuples per `Write`, and writes one `openfga_backfill_completed` audit row with the result counts.
3. **Unset `HEIMDALL_OPENFGA_BACKFILL`** (or set it to anything other than `1`) and redeploy again so subsequent boots don't repeat the scan. The job is safe to re-run — duplicate-tuple errors are caught and counted (Debug log; no audit-row noise) — but the scan is wasted work.

The backfill runner is also the recovery tool for the "Phase 3.4 step 8 reconciliation" failure mode below.

## Re-run policy

Every edit to `authz/model.fga` produces a new `AUTHORIZATION_MODEL_ID`. After any model change:

1. Merge the model edit.
2. Re-run **Step 3** of this runbook against the deployed sidecar.
3. Re-do **Step 4** with the new `AUTHORIZATION_MODEL_ID` (the `STORE_ID` is stable across model edits — it never changes).

The `STORE_ID` only changes if the `heimdall` store is deleted out-of-band (don't do that).

## Failure modes

| Symptom | Likely cause | Recovery |
| --- | --- | --- |
| `ERROR: OPENFGA_API_URL is required` | Env var unset in your shell | Export both `OPENFGA_API_URL` and `OPENFGA_PRESHARED_KEY` before running. |
| `connection refused` / sidecar unreachable | Pserv not yet healthy, or you're not on the private network | Confirm `/healthz` is `OK` on the OpenFGA service and that you're running from a Render shell or VPN tunnel. |
| `401 Unauthorized` on `fga store list` | Preshared key mismatch | Re-copy `OPENFGA_AUTHN_PRESHARED_KEYS` from the OpenFGA service into `OPENFGA_PRESHARED_KEY`. The web service env var is wired via `fromService` so it can't drift; only manual shell sessions can. |
| `openfga migrate` failed mid-flight on deploy | Network blip / Postgres unavailable / partial schema | Migrate is idempotent — retry the deploy. If it still fails, connect to `heimdall_authz` with `psql` and inspect; in the worst case `DROP DATABASE heimdall_authz` and re-run **Step 1** (no production tuples exist until **Step 3** has run). |
| `fga model write` succeeds but web service still 403s | Web service still pinned to old `OPENFGA_AUTHORIZATION_MODEL_ID` | Re-do **Step 4** with the new ID and redeploy the web service. The SDK pins the model ID for stable behaviour across revisions; it does **not** auto-discover. |
| `STORE_ID` changed unexpectedly between runs | Someone deleted the `heimdall` store | Treat as a Phase 3.4 step 8 reconciliation: re-run **Steps 3–4**, then run **Step 5** (set `HEIMDALL_OPENFGA_BACKFILL=1`, redeploy, then unset and redeploy again) to repopulate tuples from the DB. |
| Web service boots in production but OpenFGA checks all return false | One of `OPENFGA_API_URL` / `OPENFGA_STORE_ID` / `OPENFGA_AUTHORIZATION_MODEL_ID` is unset, so DI is using the `NoOp*` fallback | Set the missing env var, set `OPENFGA_HEALTH_PROBE_ENABLED=true`, and redeploy. The probe will then fail fast on subsequent misconfigurations instead of silently denying every check. |
