# Runbook — MFA enrolment + recovery

**Status:** Active. Phase 4.7 step 23.
**Source of truth:** [`docs/proposals/security-and-authorization.md`](../proposals/security-and-authorization.md) §4 (factor menu and policy stance) and §9.3 Phase 4 (steps 1–5).
**Tracked in:** [`docs/implementation/phase-4-checklist.md`](../implementation/phase-4-checklist.md) Phase 4.7 step 23.

This runbook covers operator-side MFA: first-deploy enrolment for the seed admin, handling of TOTP recovery codes, the two new rate-limit policies introduced in Phase 4.4 / 4.5, and the break-glass procedure when every admin has lost their second factor. It does **not** cover Phase 5 (JWT issuance for the API surface), WebAuthn / passkeys, or SMS OTP — those have explicit deferrals in the proposal §4 and §9.3.

## Prerequisites

- An admin user exists with `users.system_admin = true` (typically provisioned via `HEIMDALL_BOOTSTRAP_ADMIN_EMAIL` / `HEIMDALL_BOOTSTRAP_ADMIN_PASSWORD` per the [Initial admin bootstrap](../../README.md#initial-admin-bootstrap) section of the root README), and an `organization:<seed>#admin@user:<id>` tuple has been written into OpenFGA for that user (via the Phase 3.4 step 7 tuple-write hooks or the Phase 3.4 step 8 backfill in [`openfga-bootstrap.md`](./openfga-bootstrap.md)).
- The seed organisation's stable UUID is pinned via the `HEIMDALL_SEED_ORGANIZATION_ID` environment variable on the `heimdall-ticket-tracker-web` service. The policy handler keys on this UUID (not the `heimdall` slug, which is admin-renamable) — see [`src/Heimdall.Web/Bootstrap/SeedOrganizationOptions.cs`](../../src/Heimdall.Web/Bootstrap/SeedOrganizationOptions.cs) for the bound options type and the startup probe that fails fast if neither the env var nor the dev-fallback row lookup resolves.
- An authenticator app is installed on the operator's device — 1Password, Authy, Google Authenticator, or any other RFC 6238-compliant TOTP app. Hardware tokens (YubiKey OTP) are not on the Phase 4 menu (proposal §4).

## Step 1 — First-deploy enrolment for the seed admin

After Phase 4 has merged and the web service has redeployed, the seed admin must enrol on their first authenticated visit:

1. The operator browses to `/login` and submits their bootstrap email + password (the form posts to `/account/login` under the hood; `/login` is the interactive page operators can navigate to).
2. On the first navigation to any `/admin/*` route, the Phase 4.6 step 17 redirect filter (`IAuthorizationMiddlewareResultHandler`) detects that the user is an org admin without `amr=mfa` and 302-redirects to `/account/mfa/setup`.
3. `/account/mfa/setup` server-renders a fresh authenticator key (generated by `UserManager.ResetAuthenticatorKeyAsync` + `GetAuthenticatorKeyAsync`) and the corresponding `otpauth://` URI as an inline PNG via [QRCoder](https://github.com/codebude/QRCoder) (`PngByteQRCode`). No client-side JS interop is involved — the QR is a `<img src="data:image/png;base64,…">`, served under the existing `img-src 'self' data:` CSP allowance that Blazor Server already requires.
4. The operator scans the QR code in their authenticator app and submits the 6-digit code on the same page. The submit POSTs to `/account/mfa/setup/verify` (outside the interactive circuit), which calls `VerifyTwoFactorTokenAsync`, flips `users.two_factor_enabled = true`, generates 10 recovery codes via `GenerateNewTwoFactorRecoveryCodesAsync`, and writes an `mfa_enrolled` audit event in the same transaction.
5. The redirect target displays the **10 recovery codes exactly once**. They are not stored in any session or cookie longer than the single render — refreshing the page after navigating away will **not** show them again.

> **Recovery codes are hashed at rest.** `user_recovery_codes.code_hash` stores PBKDF2 hashes via the Identity password hasher; the plaintext code is never persisted and is not recoverable from a database backup. Losing all 10 recovery codes *without first disabling MFA from a logged-in session* puts the user into the "DBA-assisted reset" path described in [Step 6](#step-6--common-operator-scenarios).

## Step 2 — Recovery-code storage guidance

Treat the 10 codes the same way you would treat the master password to a vault:

- **Print** them to a sealed envelope kept in a locked drawer or safe, **or**
- **Paste** them into a password manager entry.
- Do **not** store them in the same vault entry — or even the same vault — as the operator's Heimdall account password. The whole point of a second factor is that one compromise does not equal full compromise.
- Each recovery code is single-use. After a redemption, navigate to `/account/mfa/recovery-codes/regenerate`, re-enter the account password, and reissue the full 10. The old codes are deleted server-side in the same transaction (`ReplaceCodesAsync` — see Phase 4.2 step 6).
- Phase 4.5 step 13 surfaces a one-shot warning banner — "fewer than 3 unused recovery codes remaining" — by appending `?mfaRecoveryWarning=low` to the post-challenge redirect when applicable. The expected operator response to that banner is to regenerate immediately.

## Step 3 — Rate-limit policies

Phase 4 introduces two named rate-limit policies on top of the Phase 1 `login` limiter. Both are registered in `src/Heimdall.Web/Program.cs` and attached to endpoints in [`src/Heimdall.Web/Endpoints/AccountEndpoints.cs`](../../src/Heimdall.Web/Endpoints/AccountEndpoints.cs).

| Policy | Partition key | Endpoints gated |
| --- | --- | --- |
| `mfa-setup` | `(ip, user_id)` | `POST /account/mfa/setup/verify`, `POST /account/mfa/disable`, `POST /account/mfa/recovery-codes/regenerate` |
| `mfa-challenge` | `(ip, sha256(Identity.TwoFactorUserId cookie))` | `POST /account/mfa/challenge`, `POST /account/mfa/recovery` |

`mfa-setup` covers management actions taken by an *already-authenticated* user (verifying a freshly generated secret, tearing MFA down, regenerating recovery codes). Partitioning on `(ip, user_id)` neuters brute-force guessing of the fresh-secret window without letting one IP exhaust a different user's quota.

`mfa-challenge` covers the *unauthenticated-but-mid-flow* second-factor submission. It is deliberately **separate** from the login `(ip|email)` limiter for two reasons (per Phase 4.5 step 12):

- An attacker burning their login-attempt budget cannot also burn their target's challenge budget, and vice-versa. Each layer has its own siege engine.
- Users behind a shared NAT (corporate gateway, home router, conference Wi-Fi) cannot accidentally exhaust each other's challenge budget. The 2FA pending-user cookie (`Identity.TwoFactorUserId`) is the only stable per-user handle available before the application cookie is issued; SHA-256-hashing it keeps the partition key short while avoiding leaking raw ticket bytes into logs or counter storage.

If a legitimate user trips one of these limiters (typo on a fresh authenticator binding, fat-fingered recovery code), the response is `429 Too Many Requests` with a `Retry-After` header. The Phase 1 audit-event stream already records the rejected attempts upstream of the limiter (`mfa.challenge.failed`, `mfa.recovery.failed`, etc.) — see [Step 5](#step-5--audit-event-reference).

## Step 4 — Break-glass procedure

If every org admin has simultaneously lost their MFA factor *and* their recovery codes, the system would otherwise be unrecoverable: the `RequireMfaPolicy` denies admin surfaces, and no admin can clear `users.two_factor_enabled` for themselves through the UI without first passing the challenge.

The break-glass lever (Phase 3.5 step 10 + Phase 4.6 step 16) is a one-shot operator escape hatch:

1. On the `heimdall-ticket-tracker-web` service, set `HEIMDALL_AUTHZ_BREAK_GLASS=1` and redeploy (or restart the running instance — the value is read on every request through `IOptionsMonitor`).
2. The recovering operator must already have `users.system_admin = true`. The Phase 1 bootstrap path (`HEIMDALL_BOOTSTRAP_ADMIN_*`) can be used to promote an existing user or seed a fresh one if no system admin remains.
3. The operator logs in with password only. On the next request to a `RequireMfaPolicy` surface, [`src/Heimdall.Web/Authorization/Policies/RequireMfaAuthorizationHandler.cs`](../../src/Heimdall.Web/Authorization/Policies/RequireMfaAuthorizationHandler.cs) detects both flags (`HEIMDALL_AUTHZ_BREAK_GLASS=1` **and** `users.system_admin = true`), writes an `mfa_policy_break_glass` audit event, and allows the request.
4. The operator immediately re-enrols themselves via [Step 1](#step-1--first-deploy-enrolment-for-the-seed-admin), regenerates recovery codes, and verifies that ordinary MFA-gated access works.
5. **Flip `HEIMDALL_AUTHZ_BREAK_GLASS` back to `0` (or unset it) and redeploy.** Break-glass is a temporary lever, not a configuration mode. Leaving it on means the only thing standing between a stolen password and admin-surface access is the system-admin flag — exactly the situation MFA exists to prevent.

The `mfa_policy_break_glass` audit row is the post-incident forensic anchor. Search for it after any break-glass event and write a follow-up ticket explaining why every admin lost their factor at once — the answer almost always implies a process gap worth closing.

## Step 5 — Audit-event reference

Every MFA user action emits exactly one audit row through the Phase 1 `IAuditEventWriter`. The `event_type` strings below are the **exact** literals written by [`src/Heimdall.Web/Endpoints/AccountEndpoints.cs`](../../src/Heimdall.Web/Endpoints/AccountEndpoints.cs) and [`src/Heimdall.Web/Authorization/Policies/RequireMfaAuthorizationHandler.cs`](../../src/Heimdall.Web/Authorization/Policies/RequireMfaAuthorizationHandler.cs); operator dashboards and SIEM rules should match on these literals verbatim.

| User action | `event_type` |
| --- | --- |
| Successful enrolment (TOTP verified, `two_factor_enabled` flipped) | `mfa_enrolled` |
| Successful disable (post-password-reprompt) | `mfa_disabled` |
| Enrolment: submitted TOTP rejected by `VerifyTwoFactorTokenAsync` | `mfa.enrolment.verify_failed` |
| Enrolment: `SetTwoFactorEnabledAsync(true)` reported a non-success `IdentityResult` | `mfa.enrolment.enable_failed` |
| Disable: password re-prompt failed | `mfa.disable.bad_password` |
| Disable: store-level failure after password verified | `mfa.disable.failed` |
| Login required a second factor (redirect to `/account/mfa/challenge`) | `mfa.challenge_required` |
| Challenge: TOTP accepted | `mfa.challenge.succeeded` |
| Challenge: TOTP rejected | `mfa.challenge.failed` |
| Recovery code redeemed (payload includes `remaining_count`) | `mfa.recovery.redeemed` |
| Recovery code rejected | `mfa.recovery.failed` |
| Recovery codes reissued (post-password-reprompt) | `mfa.recovery_codes.regenerated` |
| Recovery-code regeneration: password re-prompt failed | `mfa.recovery_codes.regenerate.bad_password` |
| Break-glass override taken (env var + `system_admin = true`) | `mfa_policy_break_glass` |

Snake-case event types (`mfa_enrolled`, `mfa_disabled`, `mfa_policy_break_glass`) match the older Phase 1 / Phase 2 naming; dot-separated types (`mfa.challenge.succeeded`, etc.) match the Phase 4.5 convention for sub-flow outcomes. Both shapes are intentional — see Phase 4.5 step 12's note on naming.

## Step 6 — Common operator scenarios

### User lost their authenticator device

1. User logs in with email + password.
2. On the `/account/mfa/challenge` page, they follow the "Use a recovery code" link to `/account/mfa/recovery`.
3. They redeem one of the 10 recovery codes. The redemption writes `mfa.recovery.redeemed` with `remaining_count` in the payload.
4. Once signed in, the user visits `/account/mfa/recovery-codes/regenerate`, re-enters their password, reissues the 10 codes, **and** disables and re-enrols MFA on the new device via `/account/mfa/disable` → `/account/mfa/setup`. Re-enrolment generates a fresh authenticator key, so the old device — even if recovered later — cannot produce valid codes.

### User lost both their device AND all recovery codes

There is no in-product path here by design. Plaintext recovery codes do not exist on the server, so they cannot be re-displayed. The operator-assisted path:

1. Confirm the user's identity out-of-band (the proposal §4 and §8.3 are deliberately silent on how — your environment's identity-verification policy applies).
2. During a scheduled maintenance window, a DBA flips `two_factor_enabled = false` for that user *only*:

   ```sql
   UPDATE users
      SET two_factor_enabled = false
    WHERE id = '<user uuid>';
   ```

3. Delete any leftover rows in `user_authenticator_keys` and `user_recovery_codes` for the same `user_id` (FK CASCADE on `users` makes this strictly belt-and-braces; do it anyway to keep the audit story clean).
4. Have the user log in and immediately re-enrol via `/account/mfa/setup` — same flow as [Step 1](#step-1--first-deploy-enrolment-for-the-seed-admin).
5. The DBA action is not captured by `IAuditEventWriter` (it is out-of-band SQL), so log it manually as an `mfa.dba_reset` row through whatever change-management trail you use, and reference the ticket / incident number in the payload.

This path is deliberately friction-heavy. If you find yourself running it more than once or twice a year, revisit recovery-code storage guidance ([Step 2](#step-2--recovery-code-storage-guidance)) with your operator team.
