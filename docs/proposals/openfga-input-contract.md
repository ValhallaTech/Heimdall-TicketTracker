**Status:** **Published** (2026-05-07) — input contract for [`openfga.md`](./openfga.md) Phase 3.
**Author:** Orchestrator (Copilot)
**Scope:** Heimdall.DAL (migrations + domain types + `TeamMemberRoleConverter`); read-only contract surface for Heimdall.BLL tuple-write hooks and the OpenFGA backfill job to be authored in Phase 3.
**Purpose:** Lock down — as a single, machine-readable document — the row-by-row mapping from production database columns to OpenFGA tuple shapes, so [`openfga.md`](./openfga.md) step 7 (tuple-write hooks) and step 8 (idempotent backfill job) have one unambiguous spec to read against.
**Depends on:** [`team-collaboration.md`](./team-collaboration.md) §4 step 17 (the canonical mapping) and Phase 2.4 / 2.5 / 2.6 migrations + domain types implementing it. This document is the **output** of [`team-collaboration.md`](./team-collaboration.md) Phase 2.10 step 30 and the **input** to [`openfga.md`](./openfga.md) Phase 3.4 (steps 7–8).

> This document is **research and planning only**. **No code, package, configuration, or DI changes are made in this PR.** A separate, follow-up PR (or series of PRs) will implement the design once approved.

## 1. Why this document exists

[`team-collaboration.md`](./team-collaboration.md) §4 step 17 defines, in prose, the mapping that translates Heimdall's relational hierarchy (`organizations` → `teams` → `projects` → `tickets`) and its three membership tables (`organization_members`, `team_members`, `project_members`) into OpenFGA tuples. That mapping is the only contract between Phase 2 (data) and Phase 3 (authorization).

Two consumers will read it next:

- [`openfga.md`](./openfga.md) **step 7** — tuple-write hooks in the BLL services, which must emit a tuple write for every row insert/update/delete on the source columns.
- [`openfga.md`](./openfga.md) **step 8** — the idempotent backfill / reconciliation job, which must enumerate every row in the source columns and write the equivalent tuples.

If those two consumers read the prose differently, tuples drift from rows. This document therefore restates the mapping verbatim **and** cites the production file + line range that materializes each source column, so a reviewer can confirm in one pass that the contract is implemented as written. It also calls out the two role-collapse decisions baked into the mapping (owner→admin for org/project; manager+team_lead→team#admin) so neither hook author re-derives them.

This document does **not** edit [`team-collaboration.md`](./team-collaboration.md) §4 step 17 — that document remains the prose source of truth — and it does **not** pre-empt [`openfga.md`](./openfga.md) §3 step 1 (the `model.fga` file), which expresses the same shapes in OpenFGA DSL. It is the table-shaped, citation-bearing handshake between the two.

## 3. Role-collapse decisions

Two role-collapse decisions are baked into the mapping. They are restated here verbatim from [`team-collaboration.md`](./team-collaboration.md) §4 step 17 so [`openfga.md`](./openfga.md) step 7 (hook author) and step 8 (backfill author) write them once and identically.

- **owner → admin (organizations and projects).** From [`team-collaboration.md`](./team-collaboration.md) step 17: *"(`owner` is a label distinction in the data model — \"the person who created/transferred the org\" — but at the OpenFGA layer it collapses into `admin`; without this collapse, owner rows would produce no tuple and lose all inherited permissions.)"* The data layer can serve this collapse because the `org_member_role` and `project_member_role` Postgres enums explicitly include the `'owner'` value at the row level, so the hook can read it and translate at write time — see [`M202605050013_CreateOrganizationMembers.cs`](../../src/Heimdall.DAL/Migrations/M202605050013_CreateOrganizationMembers.cs) line 27 and [`M202605050015_CreateProjectMembers.cs`](../../src/Heimdall.DAL/Migrations/M202605050015_CreateProjectMembers.cs) line 26.

- **manager + team_lead → team#admin.** From [`team-collaboration.md`](./team-collaboration.md) step 17: *"(The `manager` / `team_lead` distinction matters to §5's policy matrix but collapses at the OpenFGA layer because both have full team authority — `team_lead` is \"deputy manager\" with the same effective grants. Phase 4 may split them via a new `team#lead` relation if a real authorization difference emerges; today there is none.)"* The data layer can serve this collapse because the `team_member_role` Postgres enum and its .NET counterpart preserve both values distinctly, with a snake_case wire round-trip the hook can read deterministically — see [`M202605050014_CreateTeamMembers.cs`](../../src/Heimdall.DAL/Migrations/M202605050014_CreateTeamMembers.cs) line 26, [`TeamMemberRole.cs`](../../src/Heimdall.Core/Models/TeamMemberRole.cs) lines 17–28, and [`TeamMemberRoleConverter.cs`](../../src/Heimdall.DAL/Repositories/TeamMemberRoleConverter.cs) lines 23–34 + 47–75.

Neither collapse is reversible from the tuple side: a downstream consumer that needs to distinguish `owner` from `admin`, or `manager` from `team_lead`, must read the relational row, not the OpenFGA tuple. That is by design — the relational schema preserves the labels for product / audit purposes; the authorization layer only cares about the effective grant.

## 4. Consumers of this contract

This document is consumed by exactly two future implementations, both in [`openfga.md`](./openfga.md) Phase 3.4:

- **[`openfga.md`](./openfga.md) §3 step 7 — Tuple-write hooks.** Hooks in the BLL services emit a tuple write whenever a source row is created, updated (role change, assignee change), or deleted. Each hook reads exactly one row of §2 above and applies the §3 collapse rule when relevant.
- **[`openfga.md`](./openfga.md) §3 step 8 — Idempotent backfill / reconciliation job.** Enumerates every row in the source columns of §2 and writes the equivalent tuples; safe to re-run; doubles as the reconciliation tool against drift introduced by the direct-write consistency choice in step 7. Logs to `audit_events` per the `security-and-authorization.md` §9.3 step 2 schema.

Any future relation added to the OpenFGA model (e.g. the `ticket#watcher` future relation called out in [`openfga.md`](./openfga.md) §4 open question 4) will require a new row in §2 and a corresponding source column. Adding a relation **without** updating this document is the same drift hazard the document exists to prevent.
