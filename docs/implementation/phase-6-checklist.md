## Out of scope for Phase 6

- **Mobile / native clients.** API surface remains web-only; the bearer + refresh model Phase 5 ships is suitable for a future CLI / mobile binding, but no client lands in Phase 6.
- **CDN-hosted static assets.** `adapter-node` serves the static subtree from the same Node process per the §8 risk row in the proposal. CDN cutover is a follow-up infra change, not a Phase 6 step.
- **i18n.** English-only at cutover per §11.4 of the proposal. Adopting an i18n library (Paraglide, `svelte-i18n`) is a separate proposal.
- **WebAuthn / passkeys.** Still deferred from Phase 4 ([`phase-4-checklist.md`](./phase-4-checklist.md) "Out of scope for Phase 4"); switching frontends does not unlock it.
- **Replacing the `QRCoder` server-side PNG path.** MFA QR generation remains server-side per step 13 and the §8 risk row in the proposal.
- **Migrating `Heimdall.Web` away from `Microsoft.NET.Sdk.Web`.** May happen as a cleanup PR after step 16 but is **not** encoded as a Phase 6 step; tracked separately if and when the slim-API-host benefit becomes concrete.

## Constraints (apply to every step)

- **Version pinning is forbidden** — Renovate Bot owns versions per the repository's stored policy. The non-binding reference baselines (Svelte v5.55.7, SvelteKit 2.60.1, Vitest v4.1.6, plus the `bits-ui` / `shadcn-svelte` / Tailwind versions in scope at proposal time) appear only in the proposal narrative, never in this checklist.
- **Vitest spec authorship is owned by the JavaScript/TypeScript Unit Test Engineer agent** (`agents/javascript-unit-tests.md`) — this constraint applies to every step that ships a `*.spec.ts` file (3, 8, 9, 10, 11, 12, 13, 14, 18, 19). The Frontend Expert may only stub spec files for the test engineer to fill in.
- **Strict TypeScript** — no `any` without explicit justification, per the org JS/TS quality gate; `unknown` + narrowing is the default.
- **`async`/`await` only** — no raw Promise chains, no floating promises.
- **bUnit retirement is one-for-one with Svelte page ports** — a Razor file is removed only in the PR that lands its Svelte replacement **and** its Vitest spec, per the §6 procedural mitigation in the proposal.
- **No big-bang merges** — every step is independently committable and independently shippable per the proposal's stance.
