## Phase 6.3 — Public unauthenticated pages

> **Implementation note (coexistence).** The SvelteKit ports land **alongside** the live Razor host rather than deleting the Razor pages + bUnit tests in the same PR. The "remove Razor in the same PR" wording conflicts with this section's own "Razor pages remain reachable until the default flip in step 18" and with the §8 "two frontends running simultaneously" risk row — deleting the pages now would 404 them in production (no production proxy until step 20). Razor/bUnit retirement is therefore deferred to steps 18–19; the bUnit coverage floor is preserved meanwhile. SvelteKit is reachable through the Phase 6.2 dev `/app/*` proxy.

- [x] **8. Port unauthenticated pages — splash + errors.** Port `Splash`, `AccessDenied`, `Error`, `NotFound` to SvelteKit routes per the §7 inventory in [`blazor-to-svelte-transition.md`](../proposals/blazor-to-svelte-transition.md). During coexistence, keep the Razor pages and bUnit tests in place until the retirement window in steps 18–19; add the matching Vitest specs now (authorship owned by the JS/TS Unit Test Engineer) so parity coverage grows without a cliff. Update the step-6 reverse-proxy rule so these paths route to SvelteKit; the Razor pages remain reachable only via direct route patterns until the default flip in step 18.
- [x] **9. Port auth-entry pages.** `Login` (→ existing `POST /api/v1/auth/token`), `RegisterConfirmation`, and `ForgotPasswordConfirmation` are ported with Vitest specs. `Register`, `ForgotPassword`, and `ResetPassword` are now ported against the minimal JSON auth endpoints added in the step-9 follow-up (C# Coding + C# Unit Tests + Security Reviewer) — `POST /api/v1/auth/register`, `POST /api/v1/auth/forgot-password`, and `POST /api/v1/auth/reset-password` in `src/Heimdall.Web/Endpoints/ApiAuthEndpoints.cs` — so SvelteKit server-to-server posting no longer collides with the antiforgery-protected `/account/{register,forgot-password,reset-password}` form endpoints. The SvelteKit routes live at `Heimdall.Frontend/src/routes/register/`, `account/forgot-password/`, and `account/reset-password/`: `register/` posts to `/api/v1/auth/register` and redirects to `/register/confirmation`; `account/forgot-password/` always redirects to `/account/forgot-password/confirmation` regardless of outcome (anti-enumeration — the endpoint always returns `200 { status: "ok" }`); `account/reset-password/` posts to `/api/v1/auth/reset-password` and redirects to `/login?reset=success`. All three endpoints `.AllowAnonymous()`, return `404` when the corresponding registration / email flow is disabled, and the two password-reset routes are bound to the `password-reset` rate-limit policy. Login's MFA branch (API returns `401 { requires_two_factor: true }`) surfaces a friendly "MFA sign-in not yet available" message — the second-factor exchange lands in Phase 6.6.

## Phase 6.5 — Forms and validation (Superforms + Formsnap + Zod + FluentValidation)

> **Implementation note (retrofit, not rebuild).** This sub-phase resolves [`docs/proposals/phase-6-adr.md`](../proposals/phase-6-adr.md) §5 Open decision 3 as **Option C Hybrid: Superforms + Formsnap + Zod (client UX) + FluentValidation (.NET API trust boundary)**. The forms ported in steps 9–11 already work against the JSON API; these three steps retrofit them onto the shared forms stack **without** changing their endpoints or contracts, so each step is independently shippable and none is a big-bang rewrite. Zod gives instant browser UX only; FluentValidation is the authoritative server-side enforcement boundary — the browser is never trusted, mirroring the OpenFGA deny-closed stance.

- [ ] **13. Foundation install.** Install `sveltekit-superforms`, `formsnap`, and `zod` in `Heimdall.Frontend`; add `FluentValidation` + `FluentValidation.DependencyInjectionExtensions` to `src/Heimdall.Web`. Establish a shared Zod schema location at `Heimdall.Frontend/src/lib/schemas/` so every form derives its client shape from a single TypeScript source of truth, and register the FluentValidation validators in the Autofac `ApplicationModule` so the API resolves them through DI like every other service. Record the trust-boundary stance in `Heimdall.Frontend/README.md`: Zod = instant browser UX only; FluentValidation = authoritative server enforcement, and the API — never the browser — is the trust boundary. **Do not pin version numbers** — Renovate Bot owns package versions in the manifests. No form is migrated in this step; it only lands the dependencies, the schema directory, and the validator registration so steps 14–15 are pure retrofits. Cross-reference [`docs/proposals/phase-6-adr.md`](../proposals/phase-6-adr.md) §5.
- [ ] **14. Retrofit the auth-entry forms.** Retrofit the `Login`, `Register`, `ForgotPassword`, and `ResetPassword` routes ported in step 9 onto Superforms + Formsnap backed by shared Zod schemas under `Heimdall.Frontend/src/lib/schemas/`. The endpoints, redirect behaviour, anti-enumeration posture, and rate-limit policies from step 9 are unchanged — only the client-side form handling and validation are migrated, so the step is independently shippable. Vitest specs (JS/TS Unit Test Engineer) continue to assert role / label / a11y attributes; the Zod schema specs themselves are deferred to step 22 with the rest of the testing setup.
- [ ] **15. Retrofit the ticket forms.** Retrofit the `NewTicket` and `TicketEdit` forms ported in step 11 onto Superforms + Formsnap with shared Zod schemas under `Heimdall.Frontend/src/lib/schemas/`, **and** replace the hand-rolled backend `TryValidate` in `src/Heimdall.Web/Endpoints/ApiTicketsEndpoints.cs` with a FluentValidation `TicketDtoValidator` as the authoritative server-side enforcement boundary. The existing 422 `problem+json` field-keyed contract is preserved exactly so the SvelteKit `+page.server.ts` actions and the 422 branch from step 11 keep working unchanged. The OpenFGA `CanEditTicket` server-side checks and the Phase 3.4 step 7 tuple-write invariants flow through untouched. xUnit tests for `TicketDtoValidator` are deferred to step 22 with the rest of the testing setup.

## Out of scope for Phase 6

- **Mobile / native clients.** API surface remains web-only; the bearer + refresh model Phase 5 ships is suitable for a future CLI / mobile binding, but no client lands in Phase 6.
- **CDN-hosted static assets.** `adapter-node` serves the static subtree from the same Node process per the §8 risk row in the proposal. CDN cutover is a follow-up infra change, not a Phase 6 step.
- **i18n.** English-only at cutover per §11.4 of the proposal. Adopting an i18n library (Paraglide, `svelte-i18n`) is a separate proposal.
- **WebAuthn / passkeys.** Still deferred from Phase 4 ([`phase-4-checklist.md`](./phase-4-checklist.md) "Out of scope for Phase 4"); switching frontends does not unlock it.
- **Replacing the `QRCoder` server-side PNG path.** MFA QR generation remains server-side per step 16 and the §8 risk row in the proposal.
- **Migrating `Heimdall.Web` away from `Microsoft.NET.Sdk.Web`.** May happen as a cleanup PR after step 19 but is **not** encoded as a Phase 6 step; tracked separately if and when the slim-API-host benefit becomes concrete.

## Constraints (apply to every step)

- **Version pinning is forbidden** — Renovate Bot owns versions per the repository's stored policy. The non-binding reference baselines (Svelte v5.55.7, SvelteKit 2.60.1, Vitest v4.1.6, plus the `bits-ui` / `shadcn-svelte` / Tailwind versions in scope at proposal time) appear only in the proposal narrative, never in this checklist.
- **Vitest spec authorship is owned by the JavaScript/TypeScript Unit Test Engineer agent** (`agents/javascript-unit-tests.md`) — this constraint applies to every step that ships a `*.spec.ts` file (3, 8, 9, 10, 11, 12, 14, 15, 16, 17, 21, 22). The Frontend Expert may only stub spec files for the test engineer to fill in.
- **Zod ↔ FluentValidation parity rule** (per [`phase-6-adr.md`](../proposals/phase-6-adr.md) §9 risk table) — FluentValidation is the canonical enforcement boundary and must never be weaker than the Zod schema; a code-review checklist item enforces that any Zod schema change is mirrored in FluentValidation ("If the Zod schema changed, did the FluentValidation validator change?"). The browser is never the trust boundary.
- **Strict TypeScript** — no `any` without explicit justification, per the org JS/TS quality gate; `unknown` + narrowing is the default.
- **`async`/`await` only** — no raw Promise chains, no floating promises.
- **bUnit retirement is one-for-one with Svelte page ports** — a Razor file is removed only in the PR that lands its Svelte replacement **and** its Vitest spec, per the §6 procedural mitigation in the proposal.
- **No big-bang merges** — every step is independently committable and independently shippable per the proposal's stance.
