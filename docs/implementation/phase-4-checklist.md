## Phase 4.1 — Schema (migrations only)

- [x] **1. Migration: extend `users` with 2FA columns.** FluentMigrator. Add `two_factor_enabled bool NOT NULL DEFAULT false`. `security_stamp` already exists from Phase 1 step 1 and is re-rotated by Identity on every 2FA enable / disable / recovery-code reset, which the Phase 1 `RevalidatingServerAuthenticationStateProvider` already honours (no provider change required). pgTAP: column existence + type + default + NOT NULL coverage. See [`security-and-authorization.md`](../proposals/security-and-authorization.md) §9.3 Phase 4 step 1.
- [x] **2. Migration: `user_authenticator_keys`.** One row per user. Columns: `user_id` UUID PK FK → `users(id)` `ON DELETE CASCADE`, `provider_name` text NOT NULL (= `"Authenticator"` — the Identity built-in token-provider name; preserves the column for future provider variants), `authenticator_key` text NOT NULL (the base32 shared secret produced by `UserManager.GetAuthenticatorKeyAsync`; stored verbatim because TOTP verification requires the original secret — hashing it would break HMAC-SHA1 verification), `created_at timestamptz NOT NULL DEFAULT now()`. **Encryption-at-rest and compensating controls**: rely on PostgreSQL TDE / volume-level encryption rather than column-level Data Protection — using `IDataProtector` here would re-introduce the cross-key-ring coupling explicitly rejected in [`security-and-authorization.md`](../proposals/security-and-authorization.md) §5.4. Because the column is recoverable plaintext from a DB compromise, the runbook in step 23 must document the required compensating controls: (a) Render Postgres backups stay encrypted at rest with Render-managed keys (default), (b) `authenticator_key` is **never** read into application logs (assert via a `Heimdall.DAL.Tests` log-capture test on `HeimdallUserStore`), (c) production DB role used by `Heimdall.Web` retains only `SELECT/INSERT/UPDATE/DELETE` on `user_authenticator_keys` (no `COPY`/no superuser), and (d) any DBA-shaped access to that table is audited at the platform level. A future hardening item — out of scope here — is a managed-KMS envelope-encryption wrapper around the column; tracked as a follow-up because Heimdall does not yet have a KMS dependency. pgTAP: PK, FK CASCADE, NOT NULL invariants.
- [x] **3. Migration: `user_recovery_codes`.** Columns: `id` UUID PK, `user_id` UUID FK → `users(id)` `ON DELETE CASCADE`, `code_hash` text NOT NULL, `used_at timestamptz NULL`, `created_at timestamptz NOT NULL DEFAULT now()`. **Codes are hashed at rest** (PBKDF2 via `IPasswordHasher<HeimdallUser>` or `KeyDerivation.Pbkdf2` — never plaintext, never reversible-encrypted, never compared with `==`). Composite index `(user_id, used_at)` for the "any unused codes left?" lookup; unique constraint not required because code generation is server-side and globally random. pgTAP: PK, FK CASCADE, NOT NULL coverage; the no-plaintext-storage invariant is asserted by the step-4 / step-5 unit tests.

## Phase 4.2 — Dapper-backed Identity 2FA stores

- [x] **4. Implement `IUserTwoFactorStore<HeimdallUser>` on `HeimdallUserStore`.** Reads / writes `users.two_factor_enabled`. Re-rotates `security_stamp` on every enable / disable via `UserManager.UpdateSecurityStampAsync` (Identity wires this automatically when the store is registered through `AddIdentityCore`). xUnit coverage parity with the Phase 1 Dapper Identity stores (100% line; branch coverage on every state transition).
- [x] **5. Implement `IUserAuthenticatorKeyStore<HeimdallUser>` on `HeimdallUserStore`.** Reads / writes a single row in `user_authenticator_keys` keyed on `user_id` with `provider_name = 'Authenticator'`. Idempotent `Set` (delete-then-insert in a single transaction, **not** two separate API calls — the Phase 3 step 7 "single `Write` for assignee change" precedent applies). xUnit + Testcontainers coverage against `postgres:18-alpine` matching the Phase 1 fixture.
- [x] **6. Implement `IUserTwoFactorRecoveryCodeStore<HeimdallUser>` on `HeimdallUserStore`.** `ReplaceCodesAsync` deletes all rows for `user_id`, hashes each new code, and inserts in a single transaction. `RedeemCodeAsync` looks up by hash, sets `used_at = now()` if currently null, returns `IdentityResult.Success` on first redemption and `Failed` on reuse — atomically via `UPDATE … WHERE used_at IS NULL RETURNING id` to make the check-and-mark race-free. `CountCodesAsync` returns `COUNT(*) WHERE used_at IS NULL`. xUnit + Testcontainers covers redemption, reuse rejection, and concurrent-redeem race (two parallel `RedeemCodeAsync` calls — exactly one must succeed).

## Phase 4.3 — Identity wiring

- [x] **7. Register the 2FA stores and the TOTP token provider in `Heimdall.Web/Program.cs`.** Extend the existing `AddIdentityCore<HeimdallUser>()` call with `.AddTokenProvider<AuthenticatorTokenProvider<HeimdallUser>>(TokenOptions.DefaultAuthenticatorProvider)` and `.AddSignInManager<SignInManager<HeimdallUser>>()` (the latter is required for `TwoFactorAuthenticatorSignInAsync`; Phase 1 chose `AddIdentityCore` over `AddIdentity`, so `SignInManager` is not auto-registered). `IdentityOptions.Tokens.AuthenticatorTokenProvider` set to the default name. No JWT scheme yet — that is Phase 5. Audit-events row `event_type = 'mfa_provider_registered'` is **not** emitted (this is a startup wiring step, not a user action).
- [x] **8. Add the `RequireMfaPolicy` registration scaffold.** Register the named policy in `AddAuthorization` with a placeholder requirement that **fails closed** until step 16 wires the real handler. Document the placeholder in the policy's XML doc comment so reviewers know the gate is intentionally inert until step 16. Defence-in-depth: any page that opts into the policy before step 16 simply 403s instead of silently allowing.

## Phase 4.4 — Enrolment UI

- [x] **9. `/account/mfa/setup` Blazor SSR page (gated on the cookie scheme, not `RequireMfaPolicy`).** Generates a fresh authenticator key via `UserManager.ResetAuthenticatorKeyAsync` + `GetAuthenticatorKeyAsync`, renders the `otpauth://` URI as a server-side PNG QR code (use `QRCoder`'s `PngByteQRCode` — pure managed, no native deps, MIT-licensed). The page POSTs the user-entered 6-digit TOTP to the step-10 endpoint; on success the page advances to recovery-code display. No client-side JS interop — the QR is a server-rendered `<img src="data:image/png;base64,…">`. CSP allowance: `img-src 'self' data:` is already required by Blazor Server (§3.5 of the proposal), no new directive needed.
- [x] **10. Server-rendered POST endpoint `/account/mfa/setup/verify`.** Outside the interactive circuit, per Phase 1 step 7's precedent for login. Verifies the submitted code via `UserManager.VerifyTwoFactorTokenAsync(user, TokenOptions.DefaultAuthenticatorProvider, code)`; on success calls `SetTwoFactorEnabledAsync(true)`, generates 10 recovery codes via `GenerateNewTwoFactorRecoveryCodesAsync`, and **displays them exactly once** on the redirect target. Audit event `event_type = 'mfa_enrolled'` written in the same transaction as the `users.two_factor_enabled` flip (reusing the Phase 1 `IAuditEventWriter`). Rate-limited via a new `mfa-setup` policy on `AddRateLimiter` keyed on `(ip, user_id)` to neuter brute-force guessing of the fresh secret window.
- [x] **11. `/account/mfa/disable` Blazor SSR page + paired POST endpoint.** Requires re-entry of the current password (`CheckPasswordSignInAsync`) to short-circuit a stolen-cookie attack disabling MFA. Calls `SetTwoFactorEnabledAsync(false)` and deletes all rows in `user_authenticator_keys` + `user_recovery_codes` for the user in a single transaction. Audit event `event_type = 'mfa_disabled'`. Same `mfa-setup` rate-limit policy as step 10.

## Phase 4.5 — Challenge UI

- [x] **12. Wire the post-password challenge.** Modify the Phase 1 login POST endpoint (`/account/login`) to switch on `SignInResult.RequiresTwoFactor` (already returned by `PasswordSignInAsync` when `two_factor_enabled = true`) and redirect to `/account/mfa/challenge`. The challenge page is a server-rendered Blazor SSR form (not interactive — the second factor must not depend on a circuit being already authenticated). POST `/account/mfa/challenge` calls `SignInManager.TwoFactorAuthenticatorSignInAsync(code, isPersistent: rememberMe, rememberClient: rememberMachine)`. Audit events: `mfa.challenge_required` (on redirect), `mfa.challenge.succeeded` / `mfa.challenge.failed` (dot-separated naming convention matches the rest of `AccountEndpoints` — recovery vs. TOTP redemption are surfaced via distinct event types instead of an `is_recovery_code` payload field). Rate-limited via a new `mfa-challenge` policy keyed on `(ip, sha256(Identity.TwoFactorUserId cookie))` — separate from the login `(ip|email)` limiter so an attacker burning login attempts cannot also burn challenge attempts and vice-versa, and so users behind a shared NAT cannot exhaust each other's challenge budget. The 2FA pending-user cookie is the closest stable handle available before the application cookie is issued; hashing it keeps the partition key short without leaking ticket bytes.
- [x] **13. Recovery-code redemption path.** `/account/mfa/recovery` POST endpoint calls `SignInManager.TwoFactorRecoveryCodeSignInAsync(code)`. On success, writes audit event `mfa.recovery.redeemed` (with `remaining_count` in the payload) and, **if the user has fewer than 3 unused codes remaining**, appends `?mfaRecoveryWarning=low` to the redirect target so the next page can surface a one-shot warning with a link to `/account/mfa/recovery-codes/regenerate`. On failure writes `mfa.recovery.failed`. Same `mfa-challenge` rate-limit policy as step 12.
- [x] **14. `/account/mfa/recovery-codes/regenerate` page + POST endpoint.** Reissues 10 codes (Identity's `GenerateNewTwoFactorRecoveryCodesAsync` deletes all prior codes — `ReplaceCodesAsync` in step 6 enforces this server-side as well). Displays the new codes once and only once via the existing `IRecoveryCodeDisplayCache` from step 10. Audit event `mfa.recovery_codes.regenerated` on success; `mfa.recovery_codes.regenerate.bad_password` on a failed password re-prompt. Re-prompts for password before regeneration (same stolen-cookie defence as step 11). Bound to the `mfa-setup` rate-limit policy (this is a management action, not a challenge).

## Phase 4.6 — Admin MFA gate

- [x] **15. Capture the seed organization's stable UUID at bootstrap.** Extend `DefaultHierarchyBootstrapper` (Phase 2.3 step 9) to, after upserting the `heimdall` seed org, write its UUID into a configuration setting consumed by the step-16 policy handler. The mechanism is `IOptionsMonitor<SeedOrganizationOptions> { Guid OrganizationId }` populated at startup from one of (in order): (a) `HEIMDALL_SEED_ORGANIZATION_ID` env var (preferred for production — operator pins the value), (b) a row-lookup against `organizations WHERE slug = 'heimdall'` at startup (fallback for dev). Per the 2026-05-05 decision-log entry, **the policy must key on the UUID**, not the slug, because the `heimdall` slug is admin-renamable. Startup health probe fails fast if neither source resolves — no silent "no admins need MFA" state.
- [x] **16. `RequireMfaPolicy` handler that targets organization admins via OpenFGA.** Replace the step-8 placeholder with a real `AuthorizationHandler<RequireMfaRequirement>` that:
  1. Reads the current user's id from the `ClaimsPrincipal`.
  2. Calls `IAuthorizationService.Check("organization", seedOrgId.ToString(), "admin", userId)` through the Phase 3.3 step 6 adapter (never `OpenFgaClient` directly — same constraint as Phase 3.5 step 9). Use `consistency = HIGHER_CONSISTENCY` so a freshly-granted org-admin tuple immediately upgrades the user's MFA requirement on the next request.
  3. If the user **is** an org admin, requires `ClaimsPrincipal.HasClaim("amr", "mfa")` (set automatically by Identity on a successful TOTP / recovery sign-in) **and** `users.two_factor_enabled = true` (defence-in-depth — `amr` alone could be replayed from a session predating an MFA disable).
  4. If the user **is not** an org admin, succeeds unconditionally (regular users get the §4 "optional but encouraged" stance).
  5. Sidecar outage → deny-closed per Phase 3.5 step 10. Break-glass per the same step (`HEIMDALL_AUTHZ_BREAK_GLASS=1` + `users.system_admin = true`) emits audit event `mfa_policy_break_glass`.
- [x] **17. Redirect filter on the Blazor router.** Org admins without MFA enrolled are redirected to `/account/mfa/setup` on their next authenticated request to any policy-gated route. Implemented as a `IAuthorizationMiddlewareResultHandler` (so it composes with the existing `RedirectToLogin` from Phase 1 step 9) — never as page-by-page logic. The redirect must **not** target `/account/mfa/setup` itself (avoid loops) and must allow `[AllowAnonymous]` Phase 1 pages to keep rendering unauthenticated.
- [x] **18. Apply `RequireMfaPolicy` to admin surfaces.** Decorate every page under `/admin` (Phase 2.8 step 25) and every OpenFGA tuple-management endpoint (Phase 3.6 step 11) with `[Authorize(Policy = "RequireMfaPolicy")]`. The policy chains with the Phase 3 named policies (`CanManageMembers` et al.) via `RequireAuthorization("RequireMfaPolicy", "CanManageMembers")` — both must allow.

## Out of scope for Phase 4

- **WebAuthn / passkeys** → follow-up (proposal §9.3 Phase 4 step 5; §4 lists `Fido2NetLib` as the candidate library). Phase 4 lays the policy infrastructure WebAuthn will reuse — adding a second factor type does not require re-opening the policy or the redirect filter.
- **SMS OTP** — explicitly rejected in proposal §4 (SIM-swap, deliverability).
- **JWT / API tokens** → Phase 5.
- **Tuple-aware admin write surfaces** → Phase 6 ([`openfga.md`](../proposals/openfga.md) §4 open question 4).
- **Per-tenant MFA opt-out / opt-in policy knobs.** The §4 "optional but encouraged for regular users" stance is realised in this phase as "non-admins are not redirected"; making MFA *mandatory* for non-admins on a per-org basis is a future configuration surface, not a Phase 4 step.
- **Hardware token (YubiKey OTP non-FIDO) support.** Not on the §4 menu; revisit only if a customer requires it.
