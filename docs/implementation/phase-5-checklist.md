## Phase 5.1 — Signing keys and JWKS

- [x] **1. Migration: `signing_keys`.** FluentMigrator. Columns: `kid text PRIMARY KEY` (stable across rotations; published in JWT header + JWKS), `alg text NOT NULL` (constrained to `'RS256'` / `'ES256'` via a `CHECK` — proposal §5.1 forbids HS* / `none`), `public_jwk jsonb NOT NULL` (the serialised public half — published verbatim by step 3), `private_key_protected bytea NOT NULL` (the private half wrapped via `IDataProtector` with a dedicated purpose string `"Heimdall.JwtSigningKeys.v1"` — Data Protection is used **only** to protect the at-rest bytes, **never** to sign tokens; proposal §5.4 keeps the two key rings separate; per [`phase-5-signing-key-hardening.md`](../proposals/phase-5-signing-key-hardening.md) §2.1 the column is `bytea` rather than `text` to make the binary nature explicit and prevent encoding bugs), `created_at timestamptz NOT NULL DEFAULT now()`, `not_before timestamptz NOT NULL`, `not_after timestamptz NOT NULL`, `retired_at timestamptz NULL`. Indexes: partial `WHERE retired_at IS NULL` on `not_after` (signing-key selection hot path) and on `(not_before, not_after)` (JWKS publication). **DB-level access controls** (single atomic migration with the table create, per [`phase-5-signing-key-hardening.md`](../proposals/phase-5-signing-key-hardening.md) §2.2): create two Postgres roles — `heimdall_app` (used by the application connection) and `heimdall_signer` (table owner, used as the SECURITY DEFINER identity for signing-key operations). Apply **least-privilege column-level GRANTs** as the primary confidentiality control: `heimdall_app` holds `SELECT` on every column except `private_key_protected`, holds `UPDATE` on `retired_at` only (the soft-delete retirement path), and holds neither direct `INSERT` nor `DELETE` — new rows MUST route through the `signing_keys_insert` SECURITY DEFINER function (owned by `heimdall_signer`), and hard deletes are operator-only. `heimdall_signer` holds full `SELECT` so the `signing_keys_read_private` SECURITY DEFINER reader can return the ciphertext. Enable Row Level Security on `signing_keys` (defence in depth — column-level GRANTs are the real control) **without `FORCE`** so the SECURITY DEFINER write path (function body running as table owner `heimdall_signer`) succeeds without requiring explicit per-DML policies; ship three permissive policies — `signing_key_signer_full FOR SELECT TO heimdall_signer`, `signing_key_app_read FOR SELECT TO heimdall_app`, and `signing_key_app_update FOR UPDATE TO heimdall_app` — so the policy surface mirrors the GRANT posture exactly. Two `SECURITY DEFINER` helpers owned by `heimdall_signer` with `search_path = pg_catalog, public` — `signing_keys_insert(...)` for the ciphertext-writing INSERT and `signing_keys_read_private(text)` for the ciphertext-returning SELECT — let the single `heimdall_app` connection string call into `heimdall_signer`'s privileges only on the audited code paths; the step-22 runbook records this as the production posture. Add an `AFTER INSERT OR UPDATE OR DELETE` audit trigger on `signing_keys` writing one row to `audit_events` per state change (`token.signing_key.{generated,rotated,retired,revoked,expired}`); `INSERT` privilege on `audit_events` is granted to both `heimdall_signer` (for the SECURITY DEFINER insert path) and `heimdall_app` (for the direct `retired_at` UPDATE path) so the trigger never fails with SQLSTATE 42501. pgTAP coverage on PK, NOT NULL, the `alg` `CHECK`, the partial index, **plus** RLS-enabled-but-not-forced on the table, table owner = `heimdall_signer`, the absent `private_key_protected` column privilege for `heimdall_app`, the absent direct `INSERT` / `DELETE` for `heimdall_app`, the presence of `UPDATE(retired_at)` and absence of `UPDATE(not_after)` for `heimdall_app`, the audit trigger existence, the audit_events INSERT grants for both roles, and the `bytea NOT NULL` type assertion per step 17.
- [x] **2. `ISigningKeyService` + `SigningKeyService` in `Heimdall.BLL/Tokens`.** Constructor-injected `IDataProtectionProvider` from which the service resolves a **purpose-isolated** protector once at construction: `_keyProtector = dp.CreateProtector("Heimdall.JwtSigningKeys.v1")` (same Data Protection infrastructure wired in Phase 1 step 4 — no new dependency; the purpose string matches step 1's column documentation). Private helpers `EncryptPrivateKey(byte[] pkcs8) → byte[]` and `DecryptPrivateKey(byte[] ciphertext) → byte[]` wrap `Protect` / `Unprotect`; every plaintext PKCS#8 byte array is zeroed with `CryptographicOperations.ZeroMemory` before it leaves scope; the decrypted `RSA` / `ECDsa` object **must not** be cached in a field — it is created, used for signing, and disposed within a single method call (per [`phase-5-signing-key-hardening.md`](../proposals/phase-5-signing-key-hardening.md) §2.1 and §4 "no long-lived `SecurityKey`"). On read, `Unprotect` failure (revoked DP key ring or tampered ciphertext) surfaces `CryptographicException` unchanged — fail-closed signing is the correct posture. Methods: `GenerateAsync(SigningAlgorithm alg, TimeSpan validity)` (creates a new RSA-2048 or P-256 EC key pair via `RSA.Create` / `ECDsa.Create`, serialises the public half to JWK, wraps the private half via `EncryptPrivateKey`, inserts a row, returns the `kid` — **rejects** the call with `InvalidOperationException` if the computed overlap between the outgoing key's `not_after` and the new key's `not_before` would be shorter than `TokenOptions.SigningKeyOverlap` resolved from `IOptions<TokenOptions>`. `SigningKeyOverlap` defaults to 15 minutes and must be `>= AccessTokenLifetime` (enforced at startup by `TokenOptionsValidator` via `ValidateOnStart()`), so the in-flight-token invariant from proposal §5.4 holds without operators needing to remember it), `GetCurrentSigningKeyAsync()` (returns the newest row where `not_before <= now() < not_after AND retired_at IS NULL` — the issuer always signs with the freshest unretired key; falls back to the previous key inside the overlap window), `GetTrustedKeysAsync()` (returns every row where `now() < not_after AND retired_at IS NULL` — used by the verifier and step 3), `RetireAsync(kid)` (sets `retired_at = now()`; tokens already in flight remain verifiable until `not_after`). The proposal §5.4 90-day rotation is operationalised via a scheduled job in the step-22 runbook, not encoded into a migration. xUnit unit tests cover key generation round-trip (protect → unprotect → sign → verify), the `not_before` / `not_after` window logic, the retirement state machine, the overlap-window rejection in `GenerateAsync`, `DecryptPrivateKey` raising `CryptographicException` on a tampered ciphertext, and that two consecutive `GetCurrentSigningCredentials` calls do **not** return reference-equal `RsaSecurityKey` / `ECDsaSecurityKey` instances (no caching of the decrypted key) — 100% line and branch.
- [x] **3. `/.well-known/jwks.json` minimal-API endpoint in `Heimdall.Web`.** `[AllowAnonymous]`. Returns the JSON Web Key Set assembled from `ISigningKeyService.GetTrustedKeysAsync()` — **only the public halves**, keyed by `kid`. **Field whitelist** (per [`phase-5-signing-key-hardening.md`](../proposals/phase-5-signing-key-hardening.md) §2.3): each serialised JWK includes only `kty`, `use`, `kid`, `alg`, and the algorithm-appropriate public components (`n` + `e` for RSA; `crv` + `x` + `y` for EC); the RSA private components `d` / `p` / `q` / `dp` / `dq` / `qi` and the EC private scalar `d` must never appear in the output and this is enforced by the step-19 unit-test assertion. Response cached in the existing `IMemoryCache` with a 5-minute absolute expiration, keyed by a tag derived from the current key-ring state so `SigningKeyService` can invalidate the entry on `GenerateAsync` / `RetireAsync` (avoids per-request DB reads while keeping rotation immediately visible). `Cache-Control: public, max-age=300` (5 min) to bound verifier cache staleness without making rotation invisible — kept at 300s rather than the proposal's 3600s because the in-process key-ring invalidation already handles the staleness window. Negative `Cache-Control: no-store` is **not** used — short TTL plus the overlap window in step 2 plus the in-process invalidation is sufficient and avoids hammering the DB on every verify. Document the cache TTL and the invalidation seam in the step-22 runbook. xUnit: response shape conformance ([RFC 7517 §5](https://datatracker.ietf.org/doc/html/rfc7517#section-5)), retired keys are omitted, expired keys are omitted, the endpoint never reveals `private_key_protected` bytes (asserted via a regex on the serialised response), **and** the deserialised JWK property names for both RSA and EC keys contain none of `d` / `p` / `q` / `dp` / `dq` / `qi` per step 19.

## Phase 5.2 — Refresh-token storage

- [x] **4. Migration: `refresh_tokens`.** FluentMigrator. Columns: `id uuid PRIMARY KEY`, `user_id uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE`, `token_hash text NOT NULL UNIQUE` (**SHA-256 hex of the high-entropy random refresh token** — refresh tokens are server-generated 256-bit secrets, so a fast deterministic cryptographic hash is correct and required for the O(1) lookup that `GetByHashAsync` and the rotation `WHERE id = … AND revoked_at IS NULL` race-guard depend on; this is the standard pattern used by mainstream OAuth refresh-token stores. The recovery-code precedent in Phase 4 step 3 deliberately uses salted `IPasswordHasher` with `VerifyHashedPassword` instead of equality lookup because recovery codes are short and low-entropy; refresh tokens are neither, so the analogy does not transfer. The token plaintext is never persisted, never reversible-encrypted, and never logged), `family_id uuid NOT NULL` (a single OAuth refresh-token family; rotation links members via `parent_id` / `replaced_by`; proposal §5.1), `parent_id uuid NULL REFERENCES refresh_tokens(id) ON DELETE SET NULL` (NULL only for the family's root token), `replaced_by uuid NULL REFERENCES refresh_tokens(id) ON DELETE SET NULL` (set on rotation; the rotation `UPDATE` and the new-row `INSERT` happen in a **single transaction** — same stance as Phase 3.4 step 7's "single `Write` for assignee change"), `issued_at timestamptz NOT NULL DEFAULT now()`, `expires_at timestamptz NOT NULL` (default ~14 days sliding from issue, proposal §5.1), `revoked_at timestamptz NULL`, `revoked_reason text NULL` (constrained to `'rotated' | 'logout' | 'family_replay' | 'admin_revoke'` via `CHECK` — used by audit-event payloads and by the step-22 runbook's incident-response section). Indexes: `(user_id, family_id)` (revoke-family hot path), `(expires_at)` partial `WHERE revoked_at IS NULL` (sweeper for expired-but-not-revoked rows). pgTAP coverage on PK, FKs (CASCADE / SET NULL), NOT NULL, the unique `token_hash`, and the `revoked_reason` `CHECK`.
- [x] **5. `IRefreshTokenRepository` + `RefreshTokenRepository` in `Heimdall.DAL/Repositories`.** Dapper. Methods: `InsertAsync(RefreshToken row)`, `GetByHashAsync(string tokenHash)` (returns the row including `revoked_at` / `replaced_by` — callers need them to detect replay; the `tokenHash` argument is the deterministic SHA-256 hex from step 4, which is what makes the equality lookup correct), `RotateAsync(Guid oldId, RefreshToken newRow)` (single transaction: `UPDATE refresh_tokens SET revoked_at = now(), revoked_reason = 'rotated', replaced_by = @NewId WHERE id = @OldId AND revoked_at IS NULL RETURNING id` — the `WHERE revoked_at IS NULL` guard plus `RETURNING` makes the rotation race-free in the same shape as Phase 4.2 step 6's `RedeemCodeAsync`; if the `UPDATE` affects zero rows the caller treats it as replay and proceeds to step-10 family revocation), `RevokeFamilyAsync(Guid familyId, string reason)` (bulk `UPDATE` over the family). xUnit + Testcontainers against `postgres:18-alpine`. Covers the rotation race (two concurrent `RotateAsync` calls — exactly one must win, the loser observes the `replaced_by` link) and the family-revoke sweep. Coverage target: parity with Phase 1 / Phase 4 (100% line; branch coverage on every state transition).

## Phase 5.3 — JWT bearer scheme alongside cookie auth

- [ ] **6. Register the `JwtBearer` scheme in `Heimdall.Web/Program.cs`.** Extend the existing `AddAuthentication` chain with `.AddJwtBearer("JwtBearer", …)` — the **cookie scheme remains the default** for the Blazor SSR UI; JWT bearer is a *secondary* scheme applied only to `/api/v1/*` via `[Authorize(AuthenticationSchemes = "JwtBearer", Policy = …)]` (composed with the existing Phase 3 named policies — no parallel policy stack, proposal §9.3 Phase 5 step 6). `TokenValidationParameters`: `ValidateIssuer = ValidateAudience = ValidateLifetime = ValidateIssuerSigningKey = true`, `RequireSignedTokens = true`, `RequireExpirationTime = true`, `ClockSkew = TimeSpan.FromSeconds(30)`. Signing keys are resolved live via `IssuerSigningKeyResolver` delegating to `ISigningKeyService.GetTrustedKeysAsync()` (the bearer middleware never fetches `/.well-known/jwks.json` itself — that endpoint is for *external* verifiers; in-process the service avoids an HTTP round trip). Disable HS256 / `none` explicitly (proposal §5.1, [RFC 8725 §3.1](https://datatracker.ietf.org/doc/html/rfc8725#section-3.1)). Bound to a new `api-token` rate-limit policy keyed on `(ip, sub)` once authenticated, falling back to `(ip)` for unauthenticated traffic.
- [ ] **7. `ITokenIssuer` + `JwtTokenIssuer` in `Heimdall.BLL/Tokens`.** Issues access tokens with the configured `alg`, `kid` from `ISigningKeyService.GetCurrentSigningKeyAsync()`, and claims `sub` (user id), `email`, `amr` (mirrored from the Identity principal — `"pwd"` and, when satisfied, `"mfa"`), `mfa_enrolled` (mirrored from `users.two_factor_enabled`), `jti` (random GUID — used by step 11's denylist and the step-22 audit-event correlation), `iat`, `nbf`, `exp` (~15 min, proposal §5.1). **Does not** issue refresh tokens directly — the orchestration of access + refresh issuance lives in step 9 so the rotation transaction owns both inserts. xUnit unit tests cover claim shape, `kid` selection during a rotation overlap window, and the explicit rejection of `alg = "none"` / `HS*` even if a misconfigured row sneaks into `signing_keys`.
- [ ] **8. Audit-event types for the token lifecycle.** Constants in `Heimdall.Core` (alongside the Phase 4 MFA event names). Names use the dot-separated convention established in Phase 4.5 (`mfa.challenge.succeeded`, etc.): `token.access.issued`, `token.refresh.issued`, `token.refresh.rotated`, `token.refresh.replayed`, `token.access.revoked`, `token.refresh.family_revoked`, `token.signing_key.generated` (new key pair created by `SigningKeyService.GenerateAsync`), `token.signing_key.rotated` (a new key becomes the current signing key — outgoing key enters the overlap window), `token.signing_key.retired` (existing constant; `retired_at` set via `RetireAsync`), `token.signing_key.revoked` (emergency revocation — `not_after` forced to `now()`), `token.signing_key.expired` (a key's `not_after` passes and it is removed from the trusted set by the scheduled sweeper). The five `token.signing_key.*` constants are exactly the set written by the step-1 audit trigger and by the rotation runbook in step 22; the `signing_key.*` prefix in [`phase-5-signing-key-hardening.md`](../proposals/phase-5-signing-key-hardening.md) §2.4 is reconciled to the `token.signing_key.*` namespace already used by the merged checklist so the Phase 4 dot-separated convention is preserved unchanged. Every event written in the **same DB transaction** as its triggering state change (reusing `IAuditEventWriter` from Phase 1 step 7 — same precedent as Phase 4 step 10's `mfa_enrolled`). Payload shapes are documented inline on the constants and pinned in the step-21 acceptance test.

## Phase 5.4 — Token endpoints with refresh rotation + family-replay detection

- [ ] **9. `POST /api/v1/auth/token` — password grant.** Server-rendered minimal-API handler (outside any Blazor circuit, same precedent as Phase 1 step 7 and Phase 4 steps 10–12). Validates credentials via `SignInManager.PasswordSignInAsync(lockoutOnFailure: true)`. On `RequiresTwoFactor` the handler returns `401` with a `{"requires_two_factor": true}` body — **does not** issue tokens, and **does not** expose a separate "preauth" token; the client is expected to call the existing UI MFA challenge to obtain a cookie principal carrying `amr=mfa`, then call this endpoint a second time with that cookie attached so `SignInManager` can produce a fully-authenticated principal (this keeps the second factor in exactly one place — the existing Phase 4.5 challenge UI — instead of forking a parallel API-side TOTP path). On success: issues an access token via step 7 **and** inserts a fresh refresh-token row + family in the same transaction as the `token.access.issued` + `token.refresh.issued` audit events. Bound to the new `api-token` rate-limit policy and the existing Phase 1 `(ip|email)` login limiter (both must allow — defence-in-depth against credential stuffing).
- [ ] **10. `POST /api/v1/auth/refresh` — refresh-token rotation + family-replay detection.** Reads the refresh token from the `__Host-heimdall_refresh` cookie (proposal §5.2: refresh tokens never appear in `localStorage`; HttpOnly, `Secure`, `SameSite=Strict`, `Path=/api/v1/auth`, `__Host-` prefix). Hashes it, calls `IRefreshTokenRepository.GetByHashAsync`. **Family-replay detection logic:** if the row is missing → `401`; if `revoked_at IS NOT NULL` or `replaced_by IS NOT NULL` → this is a replay of a token that has already been used — call `RevokeFamilyAsync(family_id, 'family_replay')`, write `token.refresh.replayed` audit event with the offending `jti` + `family_id`, return `401`; if `expires_at <= now()` → `401` (the sweeper from step 4 will purge eventually). On the happy path: call `IRefreshTokenRepository.RotateAsync(oldId, newRow)` in a single transaction with `JwtTokenIssuer.IssueAccessTokenAsync` and the `token.refresh.rotated` + `token.access.issued` audit-event writes; return the new access token in the response body and the new refresh token in a refreshed `__Host-heimdall_refresh` cookie (rotation also rolls the cookie). Bound to the `api-token` rate-limit policy.

## Phase 5.5 — Redis denylist for emergency revoke

- [ ] **11. `IAccessTokenDenylist` + `RedisAccessTokenDenylist` in `Heimdall.BLL/Tokens`.** Uses the existing `StackExchange.Redis` connection registered in `Heimdall.Web/Program.cs` from Phase 3 caching — **not** a new sidecar. Keyspace `heimdall:token-denylist:<jti>`; value is the revoke reason (`'logout' | 'admin_revoke' | 'family_replay'`); TTL is the remaining access-token lifetime from `exp - now()` (so the entry self-expires exactly when the token would have, avoiding unbounded Redis growth). Methods: `DenyAsync(jti, exp, reason)`, `IsDeniedAsync(jti)`. xUnit + Testcontainers against `redis:7-alpine` covers TTL enforcement, idempotent re-deny, and the `IsDenied` round trip.
- [ ] **12. Denylist check on every `JwtBearer` request.** Wire `JwtBearerEvents.OnTokenValidated` to call `IAccessTokenDenylist.IsDeniedAsync(ctx.SecurityToken.Id)`; on hit, `ctx.Fail("denylisted")`. **Outage stance:** Redis outage is **fail-closed for admin-policy-gated endpoints** (matching Phase 3 step 10's deny-closed precedent) and **fail-open for non-admin reads** with a `token.access.denylist_unavailable` audit-event row written via `IAuditEventWriter`. The admin/non-admin distinction is made via the existing `IAuthorizationService.Check("organization", seedOrgId, "admin", userId)` adapter from Phase 3 step 6, so there is no second copy of the admin predicate. Document the trade-off in the step-22 runbook.
- [ ] **13. `POST /api/v1/auth/logout`.** Bearer-required. Reads the current access token's `jti` + `exp` from the validated principal, calls `IAccessTokenDenylist.DenyAsync(jti, exp, 'logout')`, calls `IRefreshTokenRepository.RevokeFamilyAsync(family_id, 'logout')` for the family identified by the refresh-cookie's hash, expires the `__Host-heimdall_refresh` cookie, and writes `token.access.revoked` + `token.refresh.family_revoked` audit events in a single transaction with the family revoke.

- **API client (CLI / native) refresh-token storage.** Proposal §5.2 calls out OS secret store (Keychain / Credential Manager / libsecret) as the eventual home; this phase only addresses browser callers via the `__Host-heimdall_refresh` cookie. The CLI binding lands when the first CLI surface ships.
- **Tuple-aware admin write surfaces** → Phase 7 ([`openfga.md`](../proposals/openfga.md) §4 open question 4).
- **Per-tenant API token issuance policy knobs** (e.g. disabling password-grant for specific orgs, mandating client credentials for service-to-service). Future configuration surface, not a Phase 5 step.
- **mTLS / sender-constrained tokens (RFC 8705).** Not on the §5 menu; revisit only if a customer requires it.
